# v0.10.0 — "signed evidence chain, end to end"

> Status: proposal. Synthesised 2026-04-26 from four independent v0.9.10
> evaluations (safety-critical Rust lens, Go/Python fresh-user lens,
> curious side-project lens, rmcp migration assessment) plus the
> previous fresh-eyes review and the v0.9.11 ship state.

## 1. Executive summary

v0.10.0 is the release that closes the gap between what witness *says*
on the box ("signed evidence chain for MC/DC") and what an auditor
actually receives. Three of the four v0.9.10 evaluators converged on
the same structural finding: the signed predicate carries a *branch*
summary, not the MC/DC truth table; `original_module` is null in every
envelope; and the release tarballs themselves have no provenance.
Documentation deficits compound the problem — terms like *masking*,
*unique-cause*, *ambiguous_rows*, and the wasm-vs-source polarity of
the truth table are load-bearing for a reviewer and undefined in the
docs.

The big-bang framing is deliberate. Each gap on its own is a small
fix; together they are the difference between "supplementary evidence"
and "primary evidence" in a qualification dossier (E1's
`ADOPTION-DECISION.md`). Shipping them in one numbered release lets
us re-pitch witness to the safety-critical audience with a single
upgrade story — "v0.10 signs the truth table" — instead of asking
adopters to track a v0.9.x patch chain.

Headline change a user feels: `witness verify` on a v0.10 envelope
proves the truth table, not just the branch totals; the release tarball
verifies under `cosign verify-blob`; the docs explain every field.

## 2. Synthesised findings (by theme)

### 2.1 Signed evidence chain gaps

The largest cluster, raised most rigorously in
`/tmp/witness-eval-1/FEEDBACK.md` B1–B3 and
`/tmp/witness-eval-1/BUGS.md` BUG-2/BUG-3, with reinforcement from
`/tmp/witness-eval-2/FRESH-USER-FRICTION.md` items 21–23 and
`/tmp/witness-eval-1/ADOPTION-DECISION.md` "Top 3 reasons no":

- The DSSE-signed in-toto Statement embeds branch summary
  (`total_branches`, `covered_branches`, `per_function`, `uncovered`)
  but NOT the truth table, condition pairs, or interpretation field.
  The MC/DC artefact a regulator wants signed lives unsigned in
  `report.json` next to the envelope. (E1 B1.)
- `predicate.original_module` is `null` in every envelope produced and
  in every shipped compliance bundle envelope. Signed subject is the
  *instrumented* wasm; no chain back to the source wasm. (E1 B2/BUG-3.)
- Release tarballs have no SLSA / sigstore / minisign provenance.
  `SHA256SUMS.txt` is unsigned. `gh api ... /attestations/sha256:...`
  returns 404. (E1 B3, E1 DOCS-2.)
- Schema URLs (`https://pulseengine.eu/witness-mcdc/v1`, ...
  `/witness-coverage/v1`) are referenced in every report but no
  `docs/schema/*.json` exists in the repo. (E1 F1, E1 DOCS-1.)
- `predicate.coverage.module` is an absolute filesystem path leaking
  the build host's cwd into the signed Statement. (E1 F2.)
- `measured_at` in the signed payload defeats reproducibility — same
  inputs sign to different envelopes. (E1 F3.)

### 2.2 Truth-table semantics — wasm vs source polarity

The single highest-risk reviewer-trust finding, in
`/tmp/witness-eval-1/FEEDBACK.md` B4 and `/tmp/witness-eval-1/BUGS.md`
BUG-1, with the fresh-user perspective in
`/tmp/witness-eval-2/FRESH-USER-FRICTION.md` items 9, 11, 13:

- Truth-table outcomes for the shipped fixture are reported in the
  *opposite* polarity from the source semantics. Year=2001 documents
  outcome F; report shows row 0 outcome=T. The `evaluated.{c0,c1}`
  columns are wasm-level "early-exit-fired" booleans, not source
  conditions. (E1 BUG-1.)
- No documentation explains the inversion. A reviewer cannot map the
  truth table to the source expression without prior knowledge of
  the wasm-level encoding. (E1 B4, E2 #9.)
- `trace_health.ambiguous_rows: true` on every full-MC/DC run; field
  name reads as "rows are ambiguous" but means "trace parser engaged".
  Confusing in audit artefacts. (E1 B5/BUG-11, E2 #15.)
- The wasm-level `branch 0` / `branch 1` are not mapped to the source
  short-circuit positions (first `&&` arm vs. second vs. disjunctive
  arm). Mapping is needed for requirement-to-coverage trace. (E1 F15.)
- Source has 3 conditions; report has 2 (rustc fusion). Documented
  as expected; still surprises every fresh user. (E2 #11.)

### 2.3 Documentation as load-bearing

`/tmp/witness-eval-2/EXPLAINER-AUDIT.md` is the dedicated study;
items 7-13 of `FRESH-USER-FRICTION.md` are the concrete instances;
E1 F7/F8/F10 cite the same gaps:

- No glossary. Decision / condition / branch / row / outcome /
  interpretation / masking / unique-cause / ambiguous_rows are
  undefined in any shipped doc.
- No "Rust -> wasm -> branches -> decisions -> truth table" diagram.
- `witness new` fixture is internally inconsistent on condition count
  (`lib.rs` says 2, `run.sh` says 3). (E1 F8/BUG-9.)
- Quickstart example output (`lib.rs:42`) does not match the actual
  scaffold output (`hint.rs:491`). v0.9.11 typed-args partly fixed
  this; docs are still pre-typed-args in places. (E2 #8.)
- DSSE / sigstore / cosign / in-toto / predicate / attestation used
  unexplained. (E2 #22.)

### 2.4 Tool qualification posture

Carried forward from Tier 2 (v0.9.9 CHANGELOG); reinforced by E1
B3/F1 and `LDRA-COMPARISON.md`:

- Extracted `witness-mcdc-checker` crate (~70 LoC) so consumers can
  audit the independent-effect-pair logic standalone.
- Differential testing against rustc-mcdc — "two oracles agree".
- JSON Schemas hosted at the URLs reports already cite, mirrored in
  `docs/schemas/`.
- Sigstore-OIDC release signing for the witness binary, so the
  qualification dossier has an integrity chain to the GitHub OIDC
  issuer.

### 2.5 Adoption ergonomics — `merge`, `diff`, export discovery

E1 F14/P5, original `FEEDBACK.md` items 5/7,
`LDRA-COMPARISON.md` gap #3:

- `witness merge` for multi-run aggregation. LDRA does this today;
  regulated CI typically runs 4-8 distinct test executables.
- `witness diff` for PR-shaped coverage deltas.
- Auto-discover `run_row_*` / typed exports; refuse to silently
  under-cover. v0.9.11 switched the scaffold to typed-args; we
  still need `--all-exports` or an error on un-invoked candidates.
- Source-line stability across edits (`/tmp/witness-eval/TESTS.md`
  T6: `lib.rs:42` -> `hint.rs:491` after one added export).

### 2.6 rmcp migration

`/tmp/witness-eval-rmcp/RMCP-MIGRATION.md` is explicit:
*"HAND-ROLLED for v0.9.11; revisit for v0.11+ only if a real MCP
feature need surfaces."* v0.9.11 already shipped the 17-line
`initialize` patch. No v0.10.0 work.

## 3. Proposed v0.10.0 scope

### Must ship (the big-bang headline)

1. **`witness-mcdc/v1` predicate type carrying truth tables**
   (effort: L; closes E1 B1, BUG-2, ADOPTION-DECISION re-eval gate (a)).
   New `witness predicate --kind mcdc` mode emits a Statement whose
   `predicate.mcdc` block contains the per-decision truth table,
   condition pairs with interpretation, and a sha256 of the canonical
   JSON. `witness verify` validates the inline content. Old
   `witness-coverage/v1` stays for branch-only consumers.
2. **`original_module` populated on every predicate**
   (effort: M; closes E1 B2, BUG-3). The instrument step records the
   pre-instrumentation sha256 in the manifest; predicate writes both
   subjects.
3. **Sigstore-OIDC release signing for the witness binary**
   (effort: M; closes E1 B3, DOCS-2; ADOPTION-DECISION re-eval gate
   (b)). GitHub Actions OIDC + cosign signs each release tarball;
   `.intoto.jsonl` provenance attached to the GitHub Release;
   `cosign verify-blob` example in the README.
4. **Truth-table polarity normalisation OR documented mapping**
   (effort: L; closes E1 B4, BUG-1; ADOPTION-DECISION re-eval gate
   (c)). Either the reporter emits source-equivalent outcomes (with
   a `--wasm-polarity` flag for backward compat), or the report
   includes an `interpretation_polarity: "wasm-early-exit"` header
   and the docs explain the inversion table. Pick one in design pass.
   Unclear from evals alone — flag for user; eval lens is split.
5. **JSON Schemas published at the cited URLs**
   (effort: M; closes E1 F1, DOCS-1, original Tier 2 #4). Four schemas
   in `docs/schemas/`: `witness-mcdc-v1.json`, `witness-coverage-v1.json`,
   `witness-rivet-evidence-v1.json`, `witness-trace-matrix-v1.json`.
   GitHub Pages hosting at `pulseengine.eu/witness-*/v1` URLs.
6. **Glossary + concepts page**
   (effort: M; closes E2 EXPLAINER-AUDIT cross-cutting gap, F8/F10).
   New `docs/concepts.md` with a worked walkthrough of the leap-year
   truth table row by row, plain-English definitions of every term,
   and the source -> wasm -> truth-table diagram.
7. **`witness merge`**
   (effort: L; closes E1 P5, LDRA-COMPARISON gap #3, original
   FEEDBACK #unmentioned). Combines N `run.json` files into one,
   reconciling counters; required for multi-suite CI.
8. **`ambiguous_rows` rename + field semantics doc**
   (effort: S; closes E1 B5, BUG-11). Rename to
   `trace_parser_active` (or similar) in the schema; deprecate the
   old name with a v0.10.x sunset note.

### Should ship (polish that justifies "v0.10" not "v0.9.12")

9. **`witness-mcdc-checker` extracted crate** (effort: M; original
   Tier 2 #2). ~70 LoC `find_independent_effect_pair` lifted into a
   no-deps crate consumers can audit.
10. **Differential testing against rustc-mcdc** (effort: L; original
    Tier 2 #3). CI step that compares witness's MC/DC verdict against
    rustc-mcdc on overlap fixtures; failures fail the release.
11. **`witness diff`** (effort: M; E1 P5). Emits a PR-shaped delta
    between two `run.json` files; CI gate consumer.
12. **Path stripping + `--source-date-epoch` in predicate**
    (effort: S; E1 F2, F3). Strip absolute paths to project-relative;
    honour `SOURCE_DATE_EPOCH` for reproducible envelopes.
13. **Branch-to-source-condition mapping in the report** (effort: M;
    E1 F15, E2 #11). Annotate each `branch N` with which short-circuit
    arm it represents (where DWARF + manifest can identify it).
14. **Error-message wrapping cleanup** (effort: S; E1 F5, F6, BUG-5,
    BUG-6). Strip `wasm runtime error:` from DSSE failures; wrap serde
    errors as "DSSE envelope file is malformed or truncated".
15. **`witness new --all-exports` auto-discovery** (effort: M; E1 F14,
    original FEEDBACK #5). Refuse to silently leave `run_*` exports
    un-invoked; add a discovery step that errors / warns by default.
16. **`witness viz` API stability + `/api/v1/verdicts/<n>/decisions/<id>`
    route fix** (effort: S; E1 F12, BUG-8). The README documents a
    surface that 404s; either fix the route or the docs (prefer route).

### Nice to ship (opportunistic)

17. **`witness rivet-evidence` quickstart example** (effort: S; E1 P4,
    original FEEDBACK #26). One-command example in the README.
18. **Compliance bundle de-nesting** (effort: S; E1 F13, BUG-10,
    original FEEDBACK #10). Stop the `compliance/compliance/...`
    double-nest on extraction.
19. **`seq_debug` removal from manifest** (effort: S; E1 F4, BUG-4).

21. **GitHub Action quickstart pointer** (effort: S; E1 P3). The v0.9.9

## 4. Sequencing

**Phase A — Schema work first (weeks 1-2).** Items 1, 2, 5, 8, 9.
Touches the on-disk shape that everything else depends on. Land
`witness-mcdc/v1` predicate, `original_module` plumbing, the four
JSON schemas at `docs/schemas/`, and the `ambiguous_rows` rename in
one schema-bumping wave. Extract `witness-mcdc-checker` while the
contracts are open. Dependency: nothing downstream can ship without
the predicate v2 settling.

**Phase B — Provenance + release pipeline (weeks 2-3, parallel).**
Items 3, 12. Sigstore-OIDC GitHub Actions wiring, path stripping,
`SOURCE_DATE_EPOCH` honour. Independent of Phase A schema work,
parallelisable. Dependency: needs Phase A schemas referenced by URL
to be reachable when reviewers `cosign verify-blob` and chase the
predicate type.

**Phase C — Reporter semantics (weeks 3-4).** Items 4, 13. Truth-table
polarity decision (the design pass), branch-to-condition mapping. Has
to follow Phase A because the v2 predicate is the natural place to
encode the polarity field. Highest-risk phase by far — see §6.

**Phase D — Workflow tooling (weeks 4-5).** Items 7, 11, 15. `merge`,
`diff`, auto-discovery. Independent of A/B/C from a code standpoint;
sequenced last so the schema and reporter contracts they consume are
locked.

**Phase E — Docs + polish (parallel weeks 1-5).** Items 6, 14, 16,
17-21. Glossary + concepts page MUST land *after* item 4 settles
(polarity decision is the central explainer); but the rest of the
docs work (rewrite, diagrams, fixture re-screenshot) can run in
parallel from week 1. Polish items 18-20 are last-mile.

## 7. Acceptance criteria

v0.10.0 is "done" when, on a clean macOS arm64 + Linux x86_64
download of the release tarball, all of the following are externally
verifiable:

1. `cosign verify-blob --certificate-identity-regexp ...
   witness-v0.10.0-aarch64-apple-darwin.tar.gz.sig` returns OK
   against the GitHub OIDC issuer.
2. `gh api /repos/pulseengine/witness/attestations/sha256:<tarball-sha>`
   returns a non-empty SLSA attestation.
3. `witness new && cd && ./run.sh && witness predicate --kind mcdc &&
   witness attest && witness verify` produces an envelope whose
   decoded payload contains the truth table (`predicate.mcdc.decisions[0].rows`)
   and `original_module.sha256` is non-null.
4. Tampering `report.json` after attest causes `witness verify
   --check-content` to fail with a clear "MC/DC content does not match
   signed predicate" message.
5. `curl -fsSL https://pulseengine.eu/witness-mcdc/v1` returns a JSON
   Schema document; same for the other three predicate types.
6. The `docs/concepts.md` page contains a row-by-row walkthrough of
   the leap-year truth table, definitions of masking / unique-cause /
   ambiguous_rows, and the source-to-wasm pipeline diagram.
7. `witness merge run-suite-a.json run-suite-b.json -o merged.json`
   produces a valid combined report; `witness report --input
   merged.json --format mcdc` reads it.
8. The shipped `witness new` fixture's `run.sh` comment, the
   `lib.rs` doc-comment, and the actual report agree on the condition
   count (single source of truth for the scaffold's expected output).
9. `witness verify` on a tampered envelope produces an error message
   with no `wasm runtime error:` prefix and no bare `serde error`
   wrapping.
10. `gh release view v0.10.0` shows: signed binary tarballs, signed
    `compliance-evidence.tar.gz`, predicate JSON Schemas linked,
    release notes leading with the integrity-chain story.
